# 实验1：图像数据采集

本文基于实际源代码，详细拆解 STM32 \+ RT\-Thread 环境下 OV2640 摄像头图像采集、数据传输及 PC 端接收的完整流程，重点解析资源有限场景（如 STM32F103）下的硬核优化技术，助力快速上手并规避常见坑点。

## 一、核心技术

为在 STM32F103 等资源有限的单片机上实现实时图像采集，本方案采用两大核心硬核技术，突破性能瓶颈：

- **寄存器直读**：跳过框架层封装，直接操作 GPIO 寄存器读取图像数据，最大化提升数据抓取速度；

- **中断屏蔽**：采集期间屏蔽系统调度，避免线程切换、时钟滴答等占用 CPU 资源，确保数据采集不丢帧。

## 二、硬件引脚映射

本代码强依赖特定 GPIO 物理连接，需严格对照 ov2640\.h 宏定义核对硬件连线，否则无法正常工作：

### 1\. SCCB 控制接口（软 I2C）

SCL → PB6，SDA → PB7

### 2\. 控制信号

RESET → PB9，PWDN → 未使用

### 3\. DVP 图像时钟信号

VSYNC → PA0，HREF → PB8，PCLK → PB4

### 4\. DVP 8位数据引脚（打散映射）

D0 → PB3 \| D1 → PC7 \| D2 → PC13 \| D3 → PB10
D4 → PB11 \| D5 → PB13 \| D6 → PB14 \| D7 → PB15

## 三、分步实现流程

### 第一步：软 I2C 与摄像头配置（SCCB 协议）

OV2640 的分辨率、白平衡、曝光模式及输出格式（如 JPEG），均通过 SCCB 总线（兼容 I2C）进行配置，核心实现如下：

- **软 I2C 实现（soft\_i2c\.c）**：规避硬件 I2C 时序兼容问题，利用普通 GPIO 引脚模拟 I2C 协议，实现稳定的寄存器读写；

- **寄存器配置（ov2640\.c）**：代码中包含 OV2640\_160x120\_JPEG 等参数数组（原厂提供的“魔法参数”），ov2640\_init 函数会将这些参数逐一写入摄像头寄存器，使其固定为 160x120 尺寸的 JPEG 输出模式。

### 第二步：极速裸机图像采集（核心难点）

STM32F1 系列单片机面临 PCLK 时钟频率高的问题，若使用 RT\-Thread 框架的 rt\_pin\_read 函数，CPU 无法及时抓取数据，代码采用以下策略解决（ov2640\.c → get\_jpg\_data 函数）：

1. **等待帧同步**：通过死循环等待 VSYNC 信号翻转，确认新一张图像开始输出；

2. **屏蔽系统调度**：调用 rt\_hw\_interrupt\_disable\(\)，暂停系统时钟滴答、线程切换等所有调度，让单片机专注于数据搬运；

3. **寄存器直读**：通过宏定义 OV2640\_DATA，直接读取 GPIOA/B/C 的 IDR 寄存器，结合位运算拼凑出每一个图像字节，最大化提升读取速度；

4. **寻找 JPEG 帧头尾**：边读取数据边判断帧边界，JPEG 图像以 FF D8 开头、FF D9 结尾，抓到结束标志后，立即调用 rt\_hw\_interrupt\_enable\(level\) 恢复系统调度，避免系统死机。

### 第三步：串口极速发送

采集到的 160x120 图像（约几 KB，存储在全局数组 ov2640\_framebuf 中），需通过串口快速传输至 PC 端。为保证二进制流纯净且追求最高速度，代码避开 RT\-Thread 串口驱动，手写底层串口发送逻辑（main\.c）：

```c
static void write_jpeg_data_to_uart(uint8_t *data, uint32_t len) {
    uint32_t i;
    for (i = 0; i < len; i++) {
        USART1->DR = data[i];       // 直接往数据寄存器塞数据
        while((USART1->SR & 0x40) == 0); // 死等 TXE (发送完成) 标志
    }
}
```

核心优势：无框架层冗余操作，直接操作 USART 寄存器，确保数据传输的速度和完整性。

### 第四步：Python 电脑端接收与保存

PC 端运行 ov2640\_test\.py 脚本，实现图像数据的接收、解析与保存，工作原理如下：

1. **持续监听串口**：利用 pyserial 库监听指定 COM 口，设置波特率 115200（需与单片机一致），持续读取串口字节流；

2. **提取完整 JPEG 图像**：在字节缓存中检索 JPEG 帧头（\\xff\\xd8）和帧尾（\\xff\\xd9），截取两者之间（包含头尾）的字节数据，即为完整无损的 JPEG 图像；

3. **生成图像文件**：将提取的图像数据写入文件（如 capture\_0000\.jpg），完成图像保存。

## 四、进阶思考与避坑指南

### 1\. SRAM 资源紧张问题

后续会在单片机上运行轻量级 CNN（如剪刀石头布识别），需重点关注 STM32F103 的内存限制：

- STM32F103 通常仅 64KB 或 48KB SRAM；

- 当前代码已分配 12KB 给图像缓存，加上 CNN 权重数组（占用 Flash）和计算过程中的 Tensor 缓存（占用 SRAM），内存余量极小；

- 避坑建议：坚持160x120 或更低的灰度分辨率进行 AI 推理，减少内存占用。

### 2\. 波特率瓶颈：帧率优化方向

使用 115200 波特率，传输 10KB 图像约需 1 秒钟，帧率较低，可通过以下方式优化：

- 将单片机和 Python 脚本的波特率同时提升至 460800 或更高；

- 注意：波特率提升受 USB 转串口模块性能限制，需确认模块支持对应波特率。

### 3\. 中断风险：系统兼容性注意事项

图像采集期间，中断屏蔽时间长达几十毫秒，存在以下风险及优化建议：

- 风险：若系统存在需及时响应的外部中断（如电机急停、高频通讯协议），会被严重阻塞，导致功能异常；

- 优化方向：此类视觉任务更适合使用带 DCMI 外设和 DMA 的高级单片机（如 STM32F4/H7），或专门的视觉模块，可实现无中断阻塞的高速图像采集。

## 五、总结

本方案通本方案通过“寄存器直读\+中断屏蔽”的硬核优化，解决了 STM32F103 等资源有限单片机的 OV2640 实时图像采集难题，同时实现了串口高速传输与 PC 端解析保存。需重点关注硬件引脚映射、内存占用和中断风险，根据实际需求优化波特率和硬件选型，可快速适配 AI 视觉等后续拓展场景。
